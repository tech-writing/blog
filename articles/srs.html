What is Software Requirements Specification

This article provides an intuition to build an understanding of what is a software requirements specification, and what are the benefits of having these documentation.


Overview and Benefits

The Software Requirements Specifications (SRS) is a specification for a particular software product, program, or set of programs that performs certain functions in a specific environment. Recommended Practice for Software Requirements Specifications (IEEE Std 830-1998) highlights benefits that good SRS should provide for their customers, such as the following:

  • Establish the basis for agreement between parties involved in product creation on what the software product is to do
  • Reduce the development effort
  • Provide a basis for estimating costs and schedules
  • Provide a baseline for validation and verification
  • Facilitate transfer to new customers or include new parts of the organization
  • Serve as a basis for enhancement

SRS should not describe any design or implementation details. These should be described in the design stage of the project. A requirement specifies an externally visible function or attribute of a system, while a design describes a particular subcomponent of a system or its interfaces with other subcomponents.


Outline of SRS

SRS outline, as suggested by Recommended Practice, looks like the following.

  1. Introduction
    • Purpose
    • Scope
    • Definitions, acronyms and abbreviations
    • References
    • Overview

  2. Overall description
    • Product perspective
    • Product functions
    • User characteristics
    • Constraints
    • Assumptions and dependencies

  3. Specific requirements
    • External interface requirements
    • User interfaces
    • Hardware interfaces
    • Software interfaces
    • Communications interfaces

    • Functional requirements
    • Performance requirements
    • Design constraints
    • Software system attributes
    • Other requirements

    4. Appendexies
    Index

There are some examples of organizing functional requirements: by mode, by user class, by classes or objects, by system feature, etc.

  • Functional requirements

      • ...
    • ...
    • Mode N / User class N / Class or object N / Feature N / etc.

Introduction Section of SRS

  • Introduction
    • Purpose
    • Scope
    • Definitions, acronyms and abbreviations
    • References
    • Overview

The “Purpose” defines the purpose and the intended audience for the SRS. “Scope” subsection identifies the software product to be produced by name, explains what the software product will and will not do, describes the application, including relevant benefits, objectives, and goals.

“Definitions, acronyms and abbreviations” contains the definitions of all terms required to properly interpret the SRS. This information may be provided by reference to the appendixes in the SRS or by reference to other documents.

“References” subsection provides a complete list of all documents referenced elsewhere in the SRS. This information also may be provided by reference to an appendix or to another document. “Overview” describes what the rest of SRS contains and explains how the SRS is organized.


Overall Description Section

  • Overall description
    • Product perspective
    • Product functions
    • User characteristics
    • Constraints
    • Assumptions and dependencies

“Product perspective” puts the product into perspective with other related products. If the product is independent and totally self-contained, it should be so stated here. If the product is a component of a larger system, then this subsection should relate the requirements of that larger system to functionality of the software, and should identify interfaces between that system and the software.

“Product perspective” subsection also describes how the software operates inside various constraints. For example, these constraints may include system interfaces, user interfaces, hardware interfaces, software interfaces, communications interfaces, memory, operations, site adaptation requirements. More detailed description of the interfaces should be placed in the “Specific requirements” section where interfaces are being described.

“Product functions” provides a summary of the major functions of the software without mentioning the vast amount of detail that each of those functions requires. The functions should be organized in a way that makes the list of functions understandable to the customer or to anyone else reading the document for the first time. More detailed description of the functions should be placed in the “Functional requirements” subsection of the “Specific requirements” section.

“User characteristics” subsection describes general characteristics of the intended users of the product including, for example, age, gender, country of origin, educational level, experience, and technical expertise, etc.

“Constraints” includes a general description of any other items that will limit the developers’ options. These may include regulatory policies, hardware limitations, interfaces to other applications, parallel operations, audit and control functions, higher-order language requirements, signal handshake protocols, reliability requirements, criticality of the application, safety and security considerations.

“Assumptions and dependencies” describes the factors that affect the requirements stated in the SRS. These factors are not design constraints on the software but changes to them can affect the requirements in the SRS.


Specific Requirements Section

  • Specific requirements
    • External interface requirements
    • User interfaces
    • Hardware interfaces
    • Software interfaces
    • Communications interfaces

    • Functional requirements
    • Performance requirements
    • Design constraints
    • Software system attributes
    • Other requirements

This section of the SRS should contain all of the software requirements to a level of detail sufficient to enable designers to design a system to satisfy those requirements, and testers to test that the system satisfies those requirements.

Interface descriptions (as compliment to “Product perspective” of Overall section) contain detailed descriptions of all inputs into and outputs from the software system.

“Functional requirements” defines the actions that must take place in the software to accept and process the inputs and to process and generate the outputs. These are generally listed as “shall” statements starting with “The system shall…”. These include validity checks on the inputs, exact sequence of operations, response to abnormal situations, effect of parameters.

“Performance requirements” specify both the static and the dynamic numerical requirements placed on the software or on human interaction with the software as a whole.
Static numerical requirements (also called “Capacity”) may include the number of terminals to be supported, the number of simultaneous users to be supported, amount and type of information to be handled.
Dynamic numerical requirements may include, for example, the numbers of transactions and tasks and the amount of data to be processed within certain time periods for both normal and peak workload conditions.
All of these requirements should be stated in measurable terms.

“Design constraints” specifies constraints that can be imposed by other standards, hardware limitations, etc.

Listing of “Software system attributes” may include reliability, availability, security, maintainability, portability, etc.


Conclusion

Recommended Practice for SRS provides key points, guides and insights to build intuition about SRS. As each software company has its own work specificity and culture it is a good start point to create your own corporate documentation framework that will fit your needs and projects. Potential benefits relate to implementing high standards, move from intuition to more professional ways of communication, and make projects more predictable and manageable.
